# AGENTS

本文件是 `dwebCloud` 的开发元手册，负责回答“这个项目应该如何被理解、规划、实施和交付”。

## 1. 原始愿景

`dwebCloud` 是面向应用的存储云内核，不是传统账号中心，也不是一开始就做成大而全的网盘平台。

当前阶段优先级：

- 用最小复杂度提供 app-scoped 存储空间。
- 用密钥派生身份完成认证与授权。
- 通过 `WebDAV` 为 `gaubee-2fa` 提供纯前端可接入的同步后端。
- 保持后续接入 `S3`、在线授权、更多 app 的扩展空间。

## 2. 真源分层

先读：

1. `README.md`
2. `ROADMAP.md`
3. `specs/README.md`
4. 对应模块 spec
5. 相关代码实现

职责分工：

- `ROADMAP.md`: 负责愿景收敛、阶段状态、待验收事项与当前优先级。
- `specs/`: 负责产品规格、模块边界、目标状态。
- `AGENTS.md`: 负责开发工作流、最佳实践、元规则。
- `CHAT.md`: 负责保存来自 `2FA` 拆分的原始需求轨迹。

如果冲突，优先顺序为：

1. 用户最新明确要求
2. 相关 `specs/`
3. `ROADMAP.md`
4. `AGENTS.md`
5. 当前代码现状

## 3. 开发前必做流程

1. 阅读 `ROADMAP.md`
2. 阅读 `specs/README.md` 和相关 spec
3. 阅读对应模块当前代码
4. 找出当前实现与 spec 的差距
5. 基于差距设计明确的施工计划
6. 将施工计划发给用户确认
7. 用户确认后再开始正式实现

## 4. 工程原则

- 遵循 `KISS`, `YAGNI`, `DRY`, `SOLID`
- 优先薄封装成熟组件，不重复造协议轮子
- 优先使用 Rust 在服务端与 CLI 侧收敛核心安全能力
- 当前不要扩张到完整网盘、S3 公共 API、复杂 IAM

## 5. v1 范围约束

`dwebCloud v1` 固定边界：

- backend 只实现本地文件系统
- 协议只实现 `WebDAV`
- app 默认只注册 `gaubee-2fa`
- token 通过 challenge + signature 换取
- 不实现浏览器授权页
- 不实现在线网盘 UI

## 6. 测试与交付

每次交付必须说明：

- 已验证内容
- 未验证内容
- 未验证原因

Rust 至少执行：

- `cargo test --workspace`

需要与 `2FA` 联调时，同时说明：

- `dwebCloud` 服务端版本
- `2FA` 接入方式
- 手工验证步骤
